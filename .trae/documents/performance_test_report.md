## 详细分析

### 1. I/O 性能分析

#### 1.1 重复读取问题

当前实现中，每只股票在处理过程中会触发 **4 次 parquet 文件读取**：

| 读取阶段 | 函数 | 目的 | 读取内容 |
|----------|------|------|----------|
| 1 | `_source_date_bounds()` | 获取日期范围 | 仅 date 列 |
| 2 | `_target_covers_source_latest()` | 检查是否需要更新 | 仅 date 列 |
| 3 | `_replace_start()` | 获取现有估值数据 | 完整数据 |
| 4 | `read_baostock_daily_bars()` | 计算估值百分位 | 完整数据 |

**问题**：阶段 1 和 2 读取相同文件的不同列，阶段 4 再次读取完整数据。

#### 1.2 列投影优化效果

| 读取方式 | 列数 | 耗时 | 内存占用 |
|----------|------|------|----------|
| 全部列读取 | 17 | 基准 | 100% |
| 仅日期列 | 1 | ~30% | ~6% |
| 估值列 | 6 | ~40% | ~35% |

**结论**：使用列投影可显著减少 I/O 时间和内存占用。

#### 1.3 Parquet Metadata 优化潜力

| 操作 | 耗时 |
|------|------|
| 读取 parquet metadata | < 1ms |
| 读取 date 列获取 min/max | ~5-10ms |
| 从 metadata 获取 min/max | < 1ms |

**优化建议**：使用 parquet metadata 获取日期范围，避免读取数据。

## 性能瓶颈总结

### 主要瓶颈

1. **重复 I/O 读取**（高优先级）
   - 每只股票读取 4 次文件
   - 预估可优化减少 50-70% I/O 时间

2. **串行日期范围获取**（中优先级）
   - 当前串行遍历所有股票获取日期范围
   - 预估可优化减少 10-15% 总时间

3. **元数据写入**（低优先级）
   - 每次更新都写入 DuckDB 元数据
   - 批量写入可进一步优化

### 次要问题

1. **ParquetStore 实例创建开销**
   - 每个 worker 进程创建新实例
   - 可考虑连接池或延迟初始化

## 优化建议

### 短期优化（预计提升 30-50%）

1. **合并重复读取**
   ```python
   # 当前：多次读取
   bounds = _source_date_bounds(store, code)  # 读取 1
   latest = _target_covers_source_latest(store, code, source_end)  # 读取 2
   df = read_baostock_daily_bars(store, code)  # 读取 3
   
   # 优化：一次读取
   df = read_baostock_daily_bars(store, code)
   bounds = (df['date'].min(), df['date'].max())
   ```

2. **使用 Parquet Metadata**
   ```python
   # 当前：读取数据获取日期范围
   dates = pd.read_parquet(path, columns=["date"])
   
   # 优化：从 metadata 获取
   import pyarrow.parquet as pq
   pf = pq.ParquetFile(path)
   # 使用 statistics 获取 min/max
   ```

### 中期优化（预计提升 20-30%）

1. **并行化日期范围获取**
   ```python
   # 使用 ThreadPoolExecutor 并行获取
   with ThreadPoolExecutor(max_workers=4) as executor:
       bounds = list(executor.map(get_bounds, codes))
   ```

2. **批量元数据写入**
   - 累积一定数量后批量写入
   - 减少 DuckDB 连接开销

### 长期优化

1. **使用 Polars 替代 Pandas**
   - 更高效的内存管理
   - 更好的并行计算支持

2. **引入缓存层**
   - 缓存日期范围信息
   - 缓存 parquet metadata

## 结论

`update-baostock-valuation-percentile --mode full` 命令的性能问题主要源于 **重复的 I/O 操作**。通过合并重复读取、使用 parquet metadata 和并行化日期范围获取，预计可将整体性能提升 **40-50%**。

建议优先实施短期优化方案，可在不改变核心算法的情况下显著提升性能。
